# Vim hang bugfix notes

## Symptom

Running `vim` (or any full-screen TUI that queries terminal attributes) inside
`flashback-shell-pty new` caused the live terminal to stay blank. Keystrokes
still reached vim, but the screen never refreshed.

## Root cause

`flashback-shell-pty` uses `github.com/charmbracelet/x/vt` as its internal VT
emulator. The emulator is not a one-way screen model: it maintains an internal
input pipe (`e.pr` / `e.pw`) for terminal-generated replies.

When vim starts, it queries the terminal for the current foreground/background
colors using OSC sequences such as:

```
ESC ] 10 ; ? BEL   (query foreground color)
ESC ] 11 ; ? BEL   (query background color)
```

The emulator's default handler (`vt/osc.go:handleDefaultColor`) builds a reply
and writes it to its input pipe:

```go
io.WriteString(e.pw, ansi.SetForegroundColor(xrgb.String()))
```

The original code only **wrote** child PTY output into the emulator and never
**read** from the emulator's reply pipe. Once the pipe buffer filled, the
emulator's `WriteString` blocked forever. That blocked the goroutine feeding PTY
output into the emulator, which in turn blocked vim's stdout, so vim could not
draw the screen.

## Fix

Start a dedicated goroutine that continuously reads the emulator's reply pipe
and writes those bytes back to the PTY master (`ptmx`). The replies therefore
reach vim instead of backing up.

### Code path today

`pkg/session/server.go`:

```go
// VT emulator input pipe -> PTY.
s.wg.Add(1)
go s.vtInputPipeForwarder()
```

```go
func (s *Server) vtInputPipeForwarder() {
	defer s.wg.Done()
	buf := make([]byte, 4096)
	for {
		n, err := s.vtTerm.Read(buf)
		if n > 0 {
			s.vtForwardedBytes.Add(int64(n))
			_, _ = s.ptmx.Write(buf[:n])
		}
		if err != nil {
			return
		}
	}
}
```

`pkg/vtcapture/vtcapture.go` exposes the pipe as `Read()`:

```go
// Read reads replies generated by the VT emulator (e.g. OSC color-query
// responses). These bytes should be forwarded back to the child PTY.
func (t *Terminal) Read(p []byte) (int, error) {
	return t.emu.Read(p)
}
```

So the data flow is:

```
vim -> ptmx -> vtTerm.Write() -> vt.Emulator
vt.Emulator generates reply -> vtTerm.Read() -> ptmx.Write() -> vim
```

The bytes "output" by the emulator (OSC color replies, device-attribute
replies, etc.) are read from `vtTerm.Read()` and written back to `ptmx` so the
running shell/TUI receives them.

## Secondary safeguard

The PTY reader still feeds the emulator asynchronously through a buffered
channel with drop-on-full behavior, and each `vtTerm.Write` is guarded by a
500 ms timeout. If a write hangs despite the input-pipe forwarder, the emulator
is disabled and the channel is drained so the live terminal does not freeze.

## Files involved

- `pkg/session/server.go` — `vtInputPipeForwarder()` and goroutine startup.
- `pkg/vtcapture/vtcapture.go` — `Read()` wrapper around `vt.Emulator.Read()`.
- `reference/x-main/vt/osc.go` — the emulator's default color-query reply
  handler that writes to `e.pw`.

## Takeaway

Treat the VT emulator as a bidirectional device: feed it PTY output, but also
forward its generated replies back to the PTY. Without this, query-style escape
sequences from vim and other TUIs will eventually block the output path.
